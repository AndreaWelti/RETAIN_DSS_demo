3 approcci architetturali

Scegli quello che si adatta meglio al contesto RETAIN T3.4

A

✦ Pacchetto Python a livelli + Streamlit (consigliato)

Un package retain_dss/ con moduli separati: data/, models/, optimizer/, ui/.
Streamlit come web UI per operatori. Jupyter notebook come layer di analisi che importa lo stesso package.

Pro

  • Struttura pulita — dati reali sostituiscono i sintetici senza riscrivere nulla
  • Notebook e UI condividono lo stesso codice
  • Facilmente estendibile a nuove rotte

Contro

  • Richiede più setup iniziale rispetto a semplici notebook
B

Suite di notebook Jupyter

3–4 notebook sequenziali: generazione dati → training modelli ML → ottimizzazione GA → risultati/grafici.
Nessuna UI separata: tutto si esegue in Jupyter.

Pro

  • Rapidissimo da sviluppare
  • Ottimo per presentazioni e demo narrative

Contro

  • Non fruibile da ingegneri senza Python
  • Codice difficile da riusare quando arrivano dati reali
C

Backend FastAPI + frontend

API REST Python (FastAPI) che espone endpoint di ottimizzazione. Frontend Streamlit o HTML chiama le API.
Notebook chiamano le stesse API.

Pro

  • Massimamente scalabile e deployabile
  • Adatto se in futuro il tool deve essere esposto come servizio

Contro

  • Eccessivamente complesso per una simulazione dummy
  • Introduce overhead HTTP non necessario in fase di prototipo
